### 2.3 各角色详细职责

#### @gstack:ceo（产品经理）
- **核心职责**：产品方向、需求优先级、版本决策
- **关键活动**：
  - Phase 0 主导需求规划
  - 每个 Phase 开始前确认需求范围
  - 每个 Phase 结束后进行阶段评审
  - 决定版本发布时机
  - 处理产品相关问题升级
- **关键产出**：PRD、版本规划、发布决策

#### @gstack:eng（架构师）
- **核心职责**：技术架构、性能优化、技术决策
- **关键活动**：
  - 设计系统架构和技术方案
  - 制定代码规范和技术标准
  - 参与代码审查（技术角度）
  - 性能测试和优化
  - 处理技术相关问题升级
- **关键产出**：架构设计文档、技术规范、性能报告

#### @gstack:dev（开发工程师）
- **核心职责**：功能开发、代码质量、自测
- **关键活动**：
  - 按照需求和技术规范进行开发
  - 编写单元测试和自测
  - 修复测试发现的 Bug
  - 根据审查意见修改代码
  - 配合文档工程师提供技术信息
- **关键产出**：功能代码、单元测试、代码提交

#### @gstack:test（测试工程师）
- **核心职责**：测试执行、Bug 发现、质量反馈
- **关键活动**：
  - 编写测试用例和测试计划
  - 执行功能测试和回归测试
  - 执行性能测试
  - 提交 Bug 报告并跟踪修复
  - 验证 Bug 修复
- **关键产出**：测试用例、测试报告、Bug 列表

#### @gstack:qa（QA 工程师）
- **核心职责**：质量策略、发布审核、数值平衡
- **关键活动**：
  - 制定测试策略和质量标准
  - 审核测试用例和测试报告
  - 进行发布前质量审核
  - 数值平衡和调优
  - 处理质量相关问题升级
- **关键产出**：质量报告、发布审核、数值方案

#### @gstack:docs（文档工程师）
- **核心职责**：文档编写、知识沉淀
- **关键活动**：
  - 编写技术文档和 API 文档
  - 编写用户手册和更新日志
  - 维护项目文档结构
  - 确保文档与代码同步更新
- **关键产出**：技术文档、用户手册、API 文档

#### @gstack:review（代码审查员）
- **核心职责**：代码审查、质量把关
- **关键活动**：
  - 审查所有代码提交
  - 评估代码质量和规范符合度
  - 检查安全性和性能问题
  - 给出修改建议并跟踪修改
  - 参与发布审核
- **关键产出**：审查报告、质量评分、修改建议

## 5. 质量门禁

### 5.1 各阶段准入准出标准

| 阶段 | 准入标准 | 准出标准 |
|-----|---------|---------|
| **设计阶段** | - PRD 已审批<br>- 上一版本已发布 | - 设计文档完整<br>- 设计评审通过<br>- 技术方案可行 |
| **开发阶段** | - 设计文档已审批<br>- 开发环境就绪 | - 功能开发完成<br>- 自测通过<br>- 代码已提交 |
| **测试阶段** | - 代码已提交<br>- 测试环境就绪 | - 测试用例执行率 100%<br>- 严重 Bug 数为 0<br>- 测试报告已输出 |
| **审查阶段** | - 测试已通过<br>- Bug 已修复 | - 代码审查通过<br>- 质量评分≥80<br>- 无严重问题 |
| **文档阶段** | - 代码审查通过<br>- 功能已稳定 | - 文档完整<br>- 文档评审通过<br>- 文档与代码一致 |
| **阶段评审** | - 所有交付物已完成<br>- 质量门禁已满足 | - 验收通过<br>- 发布决策已做出<br>- 下一 Phase 计划已制定 |

### 5.2 Bug 分级与处理标准

| 级别 | 定义 | 处理要求 | 负责人 |
|-----|------|---------|--------|
| **P0-致命** | 系统崩溃、数据丢失、核心功能不可用 | 必须修复，否则不能发布 | @gstack:dev |
| **P1-严重** | 主要功能异常、严重影响用户体验 | 必须修复，否则不能发布 | @gstack:dev |
| **P2-一般** | 次要功能异常、影响部分用户体验 | 建议修复，可延期到下一版本 | @gstack:dev |
| **P3-轻微** | 界面问题、文案错误、不影响功能 | 可选修复，记录到 backlog | @gstack:dev |

### 5.3 发布质量标准

| 质量指标 | 标准 | 测量方式 | 负责人 |

## 7. 问题升级机制

### 7.1 问题分类与升级路径

```
┌─────────────────────────────────────────────────────────────────┐
│                        问题升级路径                              │
└─────────────────────────────────────────────────────────────────┘

Level 1: 执行层问题
├─ 技术问题 → @gstack:eng
├─ 开发问题 → @gstack:eng
├─ 测试问题 → @gstack:qa
├─ 文档问题 → @gstack:docs
└─ 响应时间：2 小时

Level 2: 协调层问题
├─ 跨角色协作问题 → @gstack:ceo + 相关角色
├─ 资源冲突 → @gstack:ceo
├─ 进度风险 → @gstack:ceo
└─ 响应时间：4 小时

Level 3: 决策层问题
├─ 需求变更 → @gstack:ceo
├─ 范围调整 → @gstack:ceo
├─ 发布决策 → @gstack:ceo
└─ 响应时间：8 小时
```

### 7.2 问题升级标准

| 问题类型 | Level 1 处理 | Level 2 升级条件 | Level 3 升级条件 |
|---------|-------------|-----------------|-----------------|
| **技术问题** | @gstack:eng 解决 | 2 小时未解决 | 影响发布决策 |
| **开发问题** | @gstack:eng 协助 | 4 小时未解决 | 影响 Phase 完成 |
| **测试问题** | @gstack:qa 协助 | 4 小时未解决 | 影响发布决策 |
| **协作问题** | 相关角色协商 | 8 小时未解决 | 影响项目进度 |
| **需求问题** | @gstack:ceo 澄清 | 需求变更 | 范围调整 |

### 7.3 紧急问题处理

| 紧急程度 | 定义 | 响应时间 | 处理方式 |
|---------|------|---------|---------|
| **紧急** | 系统崩溃、数据丢失、无法继续开发 | 30 分钟 | 立即响应，全员参与 |
| **高** | 严重 Bug、进度严重滞后 | 2 小时 | 优先处理，相关角色参与 |
| **中** | 一般问题、可 workaround | 8 小时 | 正常流程处理 |
| **低** | 优化建议、非关键问题 | 24 小时 | 记录到 backlog |
