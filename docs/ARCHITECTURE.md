# 《我的中餐厅》v0.1 内测版 - 技术架构文档

## 1. 需求摘要

### 功能需求
- **餐厅经营**：接待顾客、赚取金币
- **装修系统**：拖拽摆放家具，自定义餐厅布局
- **厨师管理**：雇佣、升级厨师
- **菜品管理**：解锁、升级菜品

### 非功能需求
- **性能**：60FPS 流畅运行，加载时间 < 2s
- **可用性**：本地存储，离线可玩
- **数据规模**：存档 < 100KB（本地存储限制内）
- **兼容性**：微信小游戏环境（iOS/Android）

### 约束条件
- **v0.1 内测**：10-20 人体验，不正式上线
- **快速迭代**：优先开发速度，技术债可接受
- **本地存储**：无需云开发，简化部署

---

## 2. 架构决策记录 (ADR)

### ADR-001: 技术栈选择

**决策**: 微信小游戏原生开发（Canvas + JavaScript）

**状态**: 已接受

**上下文**: 
- 目标平台是微信小游戏
- 团队熟悉 Web 技术
- v0.1 需要快速验证核心玩法

**决策内容**:
- 使用微信小游戏原生 API
- 不使用游戏引擎（Cocos/Unity），减少学习成本
- 单文件 HTML 原型用于快速迭代

**后果**:
- ✅ 开发速度快，无需引擎学习
- ✅ 包体积小，加载快
- ⚠️ 复杂功能需要手写（粒子效果、物理等）
- ⚠️ 后期如需引擎，迁移成本高

---

### ADR-002: 数据存储方案

**决策**: 本地存储（wx.setStorage）

**状态**: 已接受

**上下文**:
- v0.1 不需要多人互动
- 简化部署和运维
- 内测用户数据丢失可接受

**决策内容**:
- 使用微信小游戏本地存储 API
- 存档数据结构：JSON 格式
- 自动保存 + 手动保存双机制

**后果**:
- ✅ 无需后端，开发简单
- ✅ 离线可玩
- ⚠️ 换设备数据不同步
- ⚠️ 存储空间有限（10MB）

---

### ADR-003: 架构模式

**决策**: MVC 模式（Model-View-Controller）

**状态**: 已接受

**上下文**:
- 游戏逻辑需要清晰分离
- 便于 UI 迭代
- 团队熟悉 Web 开发模式

**决策内容**:
```
Model（数据层）   →   游戏状态、存档数据
    ↓
Controller（逻辑层） →   游戏规则、业务逻辑
    ↓
View（表现层）    →   Canvas 渲染、UI 交互
```

**后果**:
- ✅ 代码结构清晰
- ✅ UI 可独立迭代
- ⚠️ 需要手动管理状态同步

## 4. 组件职责

| 组件 | 职责 | 实现方式 | 理由 |
|-----|------|---------|------|
| GameLoop | 游戏主循环 | requestAnimationFrame | 60FPS 流畅渲染 |
| Storage | 存档读写 | wx.setStorage/getStorage | 微信原生 API |

| UIManager | UI 状态管理 | 场景切换 | 简单清晰 |

## 6. 接口设计

### 游戏主循环

```javascript
// game.js
function gameLoop(timestamp) {
  update(timestamp);    // 更新逻辑
  render();             // 渲染画面
  requestAnimationFrame(gameLoop);
}
```

### 存储接口

```javascript
// storage.js
const Storage = {
  save(data) {
    wx.setStorageSync('saveData', JSON.stringify(data));
  },
  load() {
    const data = wx.getStorageSync('saveData');
    return data ? JSON.parse(data) : null;
  },
  reset() {
    wx.removeStorageSync('saveData');
  }
};
```

### 场景切换

```javascript
// scene.js
const Scene = {
  current: 'main',
  switch(sceneName) {
    this.current = sceneName;
    this.render();
  }
};
```

## 8. 非功能设计

### 性能
- **目标帧率**: 60FPS
- **加载时间**: < 2s（首屏）
- **优化策略**: 
  - 图片资源压缩
  - 对象池复用
  - 按需加载

### 可用性
- **离线可玩**: 所有资源本地缓存
- **自动保存**: 每 30 秒自动存档
- **异常恢复**: 启动时检查存档完整性

### 安全
- **存档校验**: 简单 checksum 验证
- **防作弊**: v0.1 暂不处理（内测版）

### 可扩展性
- **模块化**: 每个系统独立文件
- **配置驱动**: 数值可配置（菜品价格、升级成本等）
- **热更新**: 预留资源版本检查

## 9. 风险与应对

| 风险 | 概率 | 影响 | 应对 |
|-----|------|------|------|
| Canvas 性能瓶颈 | 中 | 中 | 限制同屏元素数量，使用对象池 |
| 存档损坏 | 低 | 高 | 存档校验 + 备份机制 |
| 微信 API 变更 | 低 | 中 | 封装 API 调用，便于适配 |
| 内存泄漏 | 中 | 中 | 定期清理，使用弱引用 |
